# 环境系统

> prime-rl Orchestrator 环境抽象层：配置、加载、运行、评估 + 新建环境脚手架。

## TL;DR

```
EnvConfig (TOML) → vf.load_environment(id) → vf.Environment
    ↓
TrainEnvs / EvalEnvs (wrappers)
    ↓
envs.start() → ZMQ servers (subprocesses)
    ↓
Training: env.run_rollout() → vf.RolloutOutput
Eval:     eval_env.evaluate() → pass@k metrics → W&B
```

---

## 1. 配置

### 基本结构

```toml
[orchestrator.train]
num_workers = 4
sampling.temperature = 0.8

[[orchestrator.train.env]]
id = "math-env"
name = "math"
sampling.max_completion_tokens = 2048

[[orchestrator.train.env]]
id = "coding-env"
ratio = 0.5  # 2:1 math:coding sampling
```

### 配置层级

```
OrchestratorConfig
├── train: TrainConfig
│   ├── env: list[TrainEnvConfig]
│   │   └── sampling: TrainSamplingConfig
│   ├── sampling: TrainSamplingConfig (组默认值)
│   └── num_workers, max_retries
├── eval: EvalConfig
│   ├── env: list[EvalEnvConfig]
│   │   └── sampling: EvalSamplingConfig
│   ├── num_examples, rollouts_per_example, interval
│   └── eval_base_model, cancel_inflight_rollouts_on_eval
```

### EnvConfig 字段

| 字段 | 类型 | 默认 | 说明 |
|------|------|------|------|
| `id` | str | "reverse-text" | verifiers env ID (可含 `@version`) |
| `name` | str\|None | None | 显示名（默认=id，必须唯一） |
| `args` | dict | {} | 传给 `vf.load_environment` |
| `address` | str\|None | None | 外部 ZMQ server（None = spawn） |
| `num_workers` | int\|"auto" | "auto" | server 进程数 |
| `ratio` | float\|None | None | 采样权重（全有或全无） |
| `max_retries` | int | 0 | 重试失败 rollout |
| `max_total_completion_tokens` | int | -1 | 多轮总 token 限制 |

### 配置验证器

1. `resolve_env_defaults` — 组默认合并到每个 env
2. `validate_unique_env_names` — resolved_name 唯一性
3. `validate_env_ratios` — ratio 全有或全无
4. `resolve_max_total_completion_tokens` — 填充 extra_env_kwargs
5. `resolve_batching` — auto num_workers 根据 max_inflight_rollouts

## 3. 服务架构

```
Main Process                    Subprocess
┌─────────────────────┐       ┌─────────────────────┐
│ TrainEnvs           │       │ ZMQEnvServer        │
│ ZMQEnvClient        │◄─TCP─┤ (env_id, num_workers)│
│                     │       │ port: 127.0.0.1:XXXX│
└─────────────────────┘       └─────────────────────┘
  await run_rollout() ───ZMQ──→ server processes
```

### 启动流程

```python
async def start():
    if config.address is None:
        # SPAWN: 子进程 + get_free_port()
        address = _spawn(...)
    else:
        # EXTERNAL: 连接已有 server
        address = config.address
    self._env_client = ZMQEnvClient(address)
    await self.env_client.wait_for_server_startup()
```

> Serial start 防止 TOCTOU port race。

## 5. 评估环境 (EvalEnv)

### evaluate() 流程

1. 预加载: `__init__` 时 `get_eval_dataset(n=num_examples).to_list()`
2. 选择模式: `requires_group_scoring` → `run_group()` 或 `run_rollout()`
3. 并发执行: `asyncio.gather(*coros)`
4. 计算指标 → 日志

### 评估指标

| 指标 | 公式 | 说明 |
|------|------|------|
| `avg@k` | mean(reward) | 主评估信号 |
| `pass@k` | Chen et al. 2021 无偏估计 | 仅用于 binary reward |
| `completion_len/mean\|std` | token 长度统计 | 调试长度问题 |
| `no_response/mean` | 空 completion 占比 | 检测 model collapse |
| `is_truncated/mean` | 截断占比 | 检测 seq_len 不够 |
| `failed_rollouts` | 失败率 | 基础设施健康度 |

```toml
[orchestrator.eval]
num_examples = 100
rollouts_per_example = 8
interval = 100  # 每 100 step 评估
```

> pass@k 只在 reward ∈ {0.0, 1.0} 时计算。

## 6. BlenderGym 环境 (多轮)

```
BlenderGymEnv (vf.MultiTurnEnv)
├── dataset.py: Task 加载
├── schema.py: Rollout/Task/Turn 结构
├── rubric.py: CLIP reward + diagnostics
└── services/: render + score (HTTP)
```

### 多轮流程

```
Turn 0: [system + goal_image + init_image + start.py] → model 生成代码
        → Parse XML → Blender 执行 → 渲染
Turn N: [上下文 + 上次渲染 + completion] → model 修改代码
        → Execute → Render
Cleanup: meta.json, trajectory.json
Reward: CLIP(final_render, goal_image)
```

## 11. 常见问题

| 问题 | 诊断 | 修复 |
|------|------|------|
| "Env not found" | id 错误或未安装 | `python -c "import env_name"` |
| 环境名冲突 | resolved_name 不唯一 | 显式设 `name` 字段 |
| 并行度不够 | num_workers=1 | auto: 1 worker/256 rollouts |
| pass@k 未计算 | reward 非 binary | 检查 rubric reward range |
| Eval 极慢 | 用了 `run_group()` | 检查 `requires_group_scoring` |
| Server 启动失败 | 端口冲突 | 设显式 `address` 或检查端口 |

---

## Gotchas

- `id = "math-env@0.1.0"` → resolved_name 会去掉 `@version` 后缀
- `envs.start()` 串行启动（避免 port race）
- `EvalEnv.__init__` 预加载所有 examples 到内存（可能很大）
- `address` 和 `num_workers` 互斥（外部 server 忽略 workers）
- `extra_env_kwargs` 自动填充，用户不应手动修改

## 附录：新建 Env 脚手架

> 从 BlenderGym 实战经验提炼的「新建 Agent RL 环境」通用模板。

### 目录结构

```
environments/<env_name>/
├── pyproject.toml              # 独立包，dep verifiers>=0.1.10
├── <env_name>/
│   ├── __init__.py             # PEP 562 lazy imports + load_environment()
│   ├── env.py                  # vf.MultiTurnEnv 子类
│   ├── rubric.py               # vf.Rubric 子类（reward + 诊断指标）
│   ├── dataset.py              # → HuggingFace Dataset
│   ├── schema.py               # 运行时状态（Task, Rollout, TurnRecord）
│   ├── prompts.py              # system prompt / instruction 模板
│   ├── artifact_manager.py     # 产物路径解析 + 保留策略
│   ├── services/               # [可选] 外部执行/评分微服务
│   └── trajectory_writer.py    # [可选] 持久化 trajectory
├── scripts/                    # benchmark 脚本
└── tests/
```

### 6 个核心模块

| 模块 | 基类 | 核心职责 |
|------|------|----------|
| **env.py** | `vf.MultiTurnEnv` | 交互循环：setup_state → get_prompt_messages → add_model_response |
| **rubric.py** | `vf.Rubric` | reward 计算 + 零权重诊断指标 + `@vf.cleanup` 写产物 |
| **dataset.py** | — | 扫描数据目录 → `Dataset.from_list([{prompt, answer, info}])` |
| **schema.py** | dataclass | `Task`(不可变) / `Rollout`(可变) / `TurnRecord` 挂在 `state["rollout"]` |
| **prompts.py** | — | 输出格式约束（影响 parse 成功率 → 间接影响 reward） |
| **services/** | FastAPI | 重量级执行/评分的 client-server 分离 + GPU 路由 + 背压 |

### Env 主体关键 hook

- `setup_state(state)` — 从 dataset info 构建 Task/Rollout，创建 work_dir
- `get_prompt_messages(state)` — turn 0 初始观察，turn N 上轮反馈
- `add_model_response(state, prompt, response)` — 解析输出 → 执行动作 → 记录
- `load_environment(**kwargs)` — 模块级工厂，verifiers 通过它发现 env

### Rubric 设计要点

- `add_reward_func(fn, weight=1.0)` — 主奖励（连续值优于 0/1）
- `add_metric(fn)` — 零权重诊断信号（format_success / exec_success）
- `@vf.cleanup` — score 之后运行，可写产物/清理磁盘

> 诊断指标极重要：训练卡住时，xml_parse_success=0.3 立刻告诉你瓶颈在格式，不用猜。

### Dataset 约定

- 列：`prompt`(list, 通常空)、`answer`(str, 通常空)、`info`(dict, env 自由空间)
- **存路径不存 payload**，base64 编码在 setup_state 时懒加载
- 支持 train/eval split（`eval_holdout` 参数）
- 数据验证在 build 阶段完成（缺文件 → 立即 fail）

### Services 何时需要

| 需要 | 不需要 |
|------|--------|
| 执行引擎启动成本高（Blender 6min JIT） | 纯 Python 计算（math verify） |
| 需要非训练 GPU（CLIP 评分） | 简单字符串匹配 |
| 需要进程隔离（sandbox 执行用户代码） | 调用外部 API（直接 httpx） |

### 从 BlenderGym 提炼的经验

| 具体教训 | 通用原则 |
|---------|---------|
| Blender 嵌入 Python 忽略 PYTHONPATH | 执行环境可能和训练环境完全隔离 |
| OPTIX 首次编译 6min | 冷启动成本要显式管理：warmup + cache 持久化 |
| S3 FUSE 不支持 rename | 存储抽象泄漏：写本地盘，异步 sync |
| 6 worker 并发 OOM | 资源隔离 + 背压：semaphore 限流 |
| VLM 模仿 prompt 中的 markdown fence | prompt 格式是 env 的一部分，影响 reward |
| 有 xml_parse_success 指标定位卡点 | 永远有阶段性诊断指标，不止 final reward |

### 新建 Env 四步法

1. **最小闭环**：dataset + env + rubric，单进程跑通一个 episode
2. **加 services**：如果执行/评分有外部依赖，抽 client/server
3. **加 artifact 管理**：保留策略、空间预算
4. **加集群 plumbing**：setup 脚本、TOML 配置、multi-worker 并发测试

> 核心问题只有三个：**agent 看到什么（prompt）→ 做什么（parser）→ 得多少分（reward）**。其他都是 infra。

### TOML 配置最小模板

```toml
[[orchestrator.train.env]]

num_workers = 4

[orchestrator.train.env.args]   # 透传给 load_environment()
```

## 补充：Rubric Reward Patterns

```python
class MyRubric(vf.Rubric):
    def __init__(self):
        super().__init__()
        self.add_reward_func(self.main_reward, weight=1.0)   # 参与训练
        self.add_metric(self.diagnostic_metric)              # 仅日志

    # Pattern 1: Exact match
    async def exact_match(self, state, info) -> float:
        return 1.0 if state["completion"].strip() == state["answer"].strip() else 0.0

    # Pattern 2: External service (如 CLIP)
    async def external_score(self, state, info) -> float:
        return await self.score_client.score(state["render"], state["goal"])

    # Pattern 3: Multi-criteria
    async def multi_criteria(self, state, info) -> float:
        correctness = 1.0 if correct else 0.0
        efficiency = 1.0 / (1.0 + turn_count)
        return 0.8 * correctness + 0.2 * efficiency

    # @vf.cleanup: 在 score_rollout 后执行（写文件、清理）
    @vf.cleanup
    async def write_artifacts(self, state) -> None:
        ...
```

## 补充：Async 数据流

```
Orchestrator
  ├─ Scheduler (维持 ~50 in-flight rollouts)
  │    └─ 连续 batching: 有 rollout 完成立即补充新的
  ├─ EnvServer (ZMQ subprocess, per-env)
  │    ├─ Worker pool (并发执行 rollout)
  │    └─ 调用 vLLM 推理 + rubric 评分
  └─ Trainer (收到 batch → 计算 advantage → 更新 θ)

Off-policy gap (k=2):
  Inference 用 π_{n-2}，Trainer 也用 π_{n-2} 产生的 rollout
  → 两者持续运行无等待
```

## 补充：Troubleshooting

| 症状 | 检查 |
|------|------|
| Reward 始终 0 | `state["completion"]` 是否为空？rubric 是否 async？parser 是否匹配？ |
| Rollout 慢 | 增大 `max_inflight_rollouts`；检查 env server 日志；增加 `num_workers` |
| Multi-turn 收敛慢 | 减 `max_turns`；加中间 reward；增大 `rollouts_per_example` |
| Buffer 卡在一个 pool | 检查 `easy_threshold/hard_threshold`；监控 `pool/*` wandb 指标 |
| 训练后 tool call 崩坏 | 确保 SFT 和 RL 的 tool call format 一致（hermes/gpt-oss） |
