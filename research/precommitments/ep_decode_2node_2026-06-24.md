## 1. Primary metric, saturation, and the comm-vs-starvation firewall

**Primary metric:** **peak sustained decode tok/s/GPU at per-engine saturation, on a pure-decode window**, normalized per GPU used.
- TP4 (A): 1 engine / 4 GPUs → rate / 4.
- EP (B/D): **sum `vllm:generation_tokens_total` counter-delta across all 4 DP engine ports**, then / 4.
- TP2+EP2 (C), TP2-noEP (E): sum across the 2 engine ports / 4.

**Exact measurement harness (no orchestrator collector, no EMA).** Enumerate engine ports at boot (TP4: 1 port; EP: ports 29520–23 → log the port→GPU map). At window-edges t0,t1 scrape each port's `/metrics`; `rate = (Σ_ports C(t1) − Σ_ports C(t0)) / (t1−t0)`. **Assert all EP ports report distinct, non-zero `generation_tokens_total`** (guards the silent 4×-undercount / label-collapse). **Cross-check:** Σ per-port gen-token deltas must be within 5% of the load generator's own returned decode-token count, else the point is void. **API-front-end check:** also record the single-endpoint aggregate rate; if single-endpoint < 0.95 × Σ-per-port, the LoRA front-end is the bottleneck → flag the point front-end-confounded (this is the production-faithful number, so flag, don't drop).

**Load driver = external open-loop client, NOT the in-repo `--bench`.** Division of labor: the **driver** is an external open-loop client (vLLM's `vllm bench serve`, or a small asyncio client hitting `/inference/v1/generate` at fixed concurrency with **real gpqa-OE debate token-ids** — never `/v1/chat/completions`, which strips `routed_experts` and would falsely pass the firewall). This resolves the v1 pad-vs-replay sub-question toward **replaying real token-ids** (long-context decode profile, not a degenerate short-context one). The **firewall + headline** come from the hand-rolled direct per-port `/metrics` counter-delta scrape with no EMA (above). The in-repo `--bench` and the `inference_metrics.py` collector are **NOT used**, for three independently disqualifying reasons: (1) no `--bench` exists on the *inference* entrypoint — only the closed-loop `orchestrator`/`rl`/`trainer`/`sft` ones; (2) `orchestrator --bench` drives the full real debate stack and **stalls after ~target_lag steps** against a standalone server (the dispatch gate reopens only when the weight-watcher advances `policy.version` — `update_dispatch_gate` `orchestrator.py:1116-1142`, `TARGET_LAG=1` `:117`, gate-call `:788`) and keeps full history (host-mem inflation); (3) the bench `output-json` schema is **trainer-side** (`mfu`, `throughput`, `step_time`, `peak_memory` — `trainer/utils.py` `export_benchmark_json`) with **no decode tok/s / no prefill-decode split** — mistaking it for the decode headline silently substitutes the wrong metric. `rl --bench` (`auto_setup_bench`) additionally injects a `FakeDataLoaderConfig` (runs on FAKE data) — not a clean decode load-generator. The `inference_metrics.py` collector smooths over a 100s window and **swallows poll failures at debug level** ⇒ silent under/over-count. **The bench `output-json` is not the headline.**

**Why not iso-concurrency (the draft's fatal contradiction).** TP4 KV ≈ 3.36M tokens (102× @32k); pure-EP KV ≈ 223k tokens/engine (6.82×/engine, G0). Matching *total in-flight* either starves EP (queues, KV-confounded) or under-saturates TP4 (decode no longer comm-bound — the 35% slice was measured at saturation). **The feasible region of "matched-concurrency ∧ both-saturated ∧ KV-clean" is EMPTY at 32k.** ∴ we drop iso-concurrency and match on **realized decode batch** instead.

**Saturation precondition (topology-neutral, decoupled from `max_num_seqs`).** A window counts only if ALL hold for ≥30s:
1. `num_requests_waiting > 0` sustained (engine always has work — offered-load saturation), AND
2. `num_requests_running` stable within ±10% (realized-batch plateau), AND
3. **iso-batch:** mean `Running`/GPU within **±15%** of the TP4 reference's mean `Running`/GPU over a matched window.

If an arm cannot hold (1)∧(2)∧(3) at a clean operating point, it yields **no point** and is reported as *"no stable iso-batch operating point at this context"* — explicitly, never silently satisfied by a starved plateau.

**Pure-decode window (kills the prefill-blend confound).** Chunked prefill interleaves prefill+decode; `generation_tokens_total` counts decode tokens whose GPU step is shared with others' prefill. **A headline window is valid only if `prompt_throughput / generation_throughput < 0.05`** (≈ pure decode). Operationally: pre-fill all requests, freeze new admission, then measure. Recurring re-prefill (a KV-starvation symptom) thus also trips this gate.

**Comm-vs-starvation firewall (replaces the static KV≤90% gate).** A point is a *clean comm measurement* iff over the window:
- **zero `vllm:num_preemptions_total` increase** (preemption = admission churn = the confound), AND
- **waiting-queue non-growing** (`d(Waiting)/dt ≈ 0`; growth ⇒ offered load exceeds service ⇒ starved batch), AND
- **tok/s CoV < 15%** within the window (G0 EP oscillated 1331→3049→1508 tok/s at ~90% KV — that thrash is the artifact).

`kv_cache_usage_perc` is **reported as a diagnostic only**, NOT the clean/dirty switch (its denominator differs across arms — TP4 pools 4 GPUs' KV, EP 1 GPU's — so a 90% cut is incomparable and biases against EP). The firewall is *preemption + stable queue + stable tok/s + pure-decode*, which is denominator-neutral.

**Operating context.** Because EP cannot reach a TP4-comparable decode batch at 32k while clean (B admits ~6/engine @32k), the **headline throughput comparison runs at 8k context** (B can sustain ~98 clean seqs @8k → an iso-batch point exists). Result generalizes to **short-context decode only**. We ALSO run the 32k point and report it: if EP cannot reach iso-batch at 32k, that is itself a **publishable kill of EP-for-long-debate**, reported as such — not hidden.

## 2. Hypotheses (LOCKED predictions + refutation thresholds)

### H_correctness — R3 router-replay index-space firewall (MUST-PASS, dominant)

**Construction.** Drive **P = 64 distinct debate-representative prompts** (real gpqa-OE debate prompt token-ids; *prompt diversity over tokens-per-prompt* because cells are autocorrelated — see sample-size note), greedy `temperature=0`, fixed `seed=0`, `max_tokens=512`, `ignore_eos=true`, **`enable_prefix_caching=false`**, via `/inference/v1/generate`. Pull `routed_experts` (b64, shape `(seq_len, 40, 8)`, uint8 global IDs). **Three legs:**

1. **Trainer leg (THE firewall — closes the inference→trainer index space).** Run ONE trainer forward in eval mode (no opt step) with route capture on the *same* greedy prompts/token-ids; compare the trainer's own `torch.topk(scores)` top-8 set per (token,layer) against the **TP4 (A)** captured set. *This is the only leg that certifies captured-ID-i indexes the same physical expert the trainer's gate-column-i indexes (after LoRA expert-MLP sharding).* It catches **common-mode** corruption (offset/permuted logical space, off-by-one/BOS misalignment) that EP-vs-TP4 is blind to. Call its divergence `d_trainer`.
2. **EP-vs-TP4 leg.** Compare EP (B) top-8 set vs TP4 (A) per (token,layer). Call it `d_EPvsTP4`. This only needs to certify *EP adds no divergence beyond the trainer leg*.
3. **Alignment spot-check (cheap, by-eye).** On 2–3 prompts pick a token with distinctive routing; verify the captured top-8 is plausible and **shifts when the token shifts** (manual off-by-one / `routed_experts_prompt_start` check the automated diff can't do).

**Per-layer gating (not global mean).** Aggregating over 40 layers hides a single corrupt layer (one fully-corrupt layer ≈ 2.5% global; a sub-layer corruption sinks under any floor). ∴ gate **per layer L∈0..39 separately**, and stratify by router family: report **GatedDeltaNet (30 layers) vs full-attn (10 layers)** divergence vectors separately (their router numerics differ).

**Null floor calibration (run FIRST, fixes the inert-null bug).** At greedy+seed0 a "two-seed TP4" null is **inert** (seed only affects sampling, which is off) → it collapses to ~0 and pins τ to the floor with no cross-layout content. Instead:
- `d_EPvsEP` = two EP DP engines, same prompts (the *correct-but-different reduction-order* floor) → **sets τ**.
- `d_TP4xTP4` = **two separate TP4 engine instances** (different NCCL init / GPU sets), greedy → cross-instance reduction-nondeterminism floor (sanity, expect tiny).

**Margin diagnostic (model-scale-calibrated, not hardcoded nats).** Gate logits pass `F.softmax` in fp32 (`moe.py:683`), and Qwen3.5's gate-logit scale is unverified — so 0.05/0.20 *raw-nat* constants are not portable. Replace with two data-driven discriminators computed from the null run:
- **Normalized gap** `ĝ = (logit_8th − logit_9th)/std(logits_token)`. Small-margin cutoff `c_small` = **95th percentile of ĝ over the `d_EPvsEP` divergent cells** (locked from the null, before the EP-vs-TP4 diff).
- **EP's rank-of-TP4's-pick** (the sharp discriminator): for each divergent cell, where does EP rank the expert TP4 chose? **Benign ⇒ rank 9–12 (just-missed); corruption ⇒ rank ≫ (e.g. ≥20).**

**Statistical test (replaces the hand-picked 3× multiplier; cluster-robust).** Cells are autocorrelated → effective n ≈ prompts×layers, not 1e7. Per layer L: test whether `d_EPvsTP4[L]` exceeds `d_EPvsEP[L]` via a **two-proportion test with prompt-clustered (block-bootstrap over the 64 prompts) CIs, α=0.01**. The **0.5% absolute floor stays** as a convention so a near-zero null doesn't make the test unattainably tight.

**LOCKED pass rule — ALL must hold:**
1. **Trainer leg:** `d_trainer[L] ≤ max(d_EPvsEP[L], 0.5%)` for **every** L, AND no structural signature (below), AND alignment spot-check passes. *(If the trainer leg fails, EP is irrelevant — the whole R3 path is corrupt; report and stop.)*
2. **EP leg:** for every L, `d_EPvsTP4[L]` is **not significantly greater** than `d_EPvsEP[L]` (clustered two-proportion, α=0.01), with the 0.5% floor.
3. **Margin concentration (per layer):** ≥90% of divergent cells have `ĝ < c_small`; EP's rank-of-TP4's-pick is 9–12 for ≥90% of divergent cells; divergence restricted to `ĝ > 2·c_small` does not exceed `d_EPvsEP`.
4. **No structural signature:** divergent EP IDs are NOT a deterministic function of TP4 IDs (check `EP_id == TP4_id mod 64`, `±k·64`, per-rank-block permutation → local-rank-relative capture = **corrupt**).
5. **Sanity asserts (lock-time):** captured `top_k == 8`; captured shape `== (seq_len, L_model, 8)` with `L_model` read from model config (not hardcoded 40); `trainer.config.num_experts == inference.num_experts == 256` (logged both); serialized dtype string `uint8` matches both ends; **router subclass identical across A and EP arms** (boot-log assert, Section 0 caveat).

**Prediction (confidence 75%, down from 80%):** pre-EPLB capture on a ReplicatedLinear gate ⇒ `d_trainer ≈ d_EPvsTP4 ≈ d_EPvsEP` (all small), margins concentrate, no structural signature → **PASS**. (Lowered because the base_router-vs-fused_moe_router subclass identity and the capture-buffer-assembly branch — naive-dispatch cumsum vs SP all_gather, which provably *differ* across arms — are unverified until boot.)

**Refutation / ABORT-EP signature (any one):** trainer leg fails on any L (R3 corrupt regardless of EP); OR EP leg significantly exceeds `d_EPvsEP` on any L; OR margin/rank-of-pick shows corruption-shaped divergence (large ĝ, or EP ranks TP4's pick ≫12); OR a structural permutation signature; OR a sanity assert trips (`top_k≠8`, shape/dtype/num_experts mismatch, divergent router subclass). Any ⇒ **EP routing is corrupt under R3; abort all EP RL; EP throughput numbers are UNUSABLE FOR RL.**

**Locked non-goal:** the gate certifies expert **IDENTITY**, not gating-**weight** fidelity. The trainer re-derives `top_scores` from its own `scores` (`moe.py:692`), so weight drift is bounded by hidden-state drift, which the margin diagnostic proxies. The off-policy-weight channel is **unverified by design**. Also unverified: prefix-cache-ON / stochastic / 16k-token *production-regime* capture equivalence (this gate runs deliberately-benign greedy/no-cache/short conditions) — the 50-step RL confirmation run owns that (Section 5.7).

### H_KV_confound — disentangle comm-benefit from KV-penalty (clean attribution arms)

**Arm-confound table (verified):** Arm **C (TP2+EP2)** changes TWO vars vs B (re-adds a 2-GPU all-reduce AND ~2× KV) → a C-win is NOT attributable to comm. Arm **D (EP+fp8KV)** changes KV dtype, which also halves KV-read DRAM traffic (decode is partly memory-bound, DRAM read 97% burst) → a D-win could be DRAM-bw not comm. ∴ **comm-reclaim is credited ONLY via a controlled subtraction:**
- **E (TP2, EP OFF)** isolates the TP-group-size effect → subtract from C to recover the EP-all2all contribution.
- **F (TP4 + fp8KV)** isolates the fp8-KV-DRAM effect → subtract from D to recover the comm-with-KV-relief contribution.

**LOCKED prediction (confidence 55%):** comm benefit is **established** iff `(D − F) > 0` beyond noise at iso-batch — i.e. EP+fp8KV beats TP4+fp8KV with the fp8 effect netted out. C is a **cross-check only**: `(C − E) > 0` corroborates.
**Refutation:** `(D − F) ≤ 0` at iso-batch ⇒ EP comm reclaim NOT established (apparent EP gains were KV/DRAM/batch artifacts).
**Contingency (pre-stated):** if **D is invalidated** (fp8 no-op or GDN page-size error, Section 3) then F's subtraction partner is gone ⇒ **comm-attribution is declared UNTESTABLE this run**; the experiment answers H_throughput-with-residual-confound only. We do **not** back-fill attribution from C alone (C confounds comm with TP-group-size).

**Minimum-detectable-effect (interpretable null).** Given ≥3 launches and the realized cross-launch SD, compute MDE; a null is reported as *"comm reclaim < MDE% (underpowered below this)"*, never as bare "negligible."

### H_expert_skew — per-rank dispatch imbalance ⇒ local expert-GEMM straggler (offline diagnostic; refines H_throughput kill-attribution)

**In scope, ZERO extra runs.** Computed offline from the *same* `routed_experts` capture the H_correctness firewall already pulls (**arm B only** — see source note below; shape `(seq_len, 40, 8)`, uint8 global logical IDs). Expert-load skew is a decode-physics property. **Mechanism (red-team-corrected for the locked backend):** under the locked `all2all_backend='allgather_reducescatter'`, dispatch is `all_gatherv` and combine is `reduce_scatterv`, both sized by `get_chunk_sizes_across_dp_rank()` — i.e. by per-DP-rank **token** counts, which are *symmetric* across ranks (every rank gathers the full token superset; non-local experts are masked in the local grouped GEMM). **Therefore expert-load skew does NOT change the all2all collective volume.** What it changes is the **local expert-MLP grouped-GEMM compute**: the rank that owns the most-dispatched experts does the most GEMM work, and because that local compute is followed by the **synchronous `reduce_scatterv` combine barrier**, the whole decode step waits on the **expert-GEMM straggler rank**. So skew ⇒ compute-straggler bounded by the combine barrier, NOT "all2all latency ∝ max-rank-load." (Verified: vLLM 0.22 `.venv/.../device_communicators/all2all.py:84-136` `AgRsAll2AllManager` dispatch/combine; masking in fused_moe local GEMM.) This still explains a *potential pure-EP throughput LOSS* and splits a kill into recoverable-vs-intrinsic — and EPLB is precisely the expert→rank rebalancing lever for this compute-straggler, so the diagnostic's EPLB-relevance survives the mechanism correction; only the named kernel changed.

**PLACEMENT (locked, verified — re-confirmed at boot):** tp1/dp4, `enable_eplb=false`, `all2all_backend='allgather_reducescatter'` ⇒ **LINEAR contiguous 64-blocks** (vLLM 0.22 `.venv/.../fused_moe/expert_map_manager.py:67-79`: `base_experts = 256//4 = 64`, `remainder=0`, linear branch `start_idx = ep_rank*64`; default strategy `'linear'` `parallel.py`). `round_robin` is **unreachable here** — `determine_expert_placement_strategy` (`expert_map_manager.py:123-147`) falls back to `linear` because `round_robin` requires `num_expert_group > 1` (Qwen3.5 sets `num_expert_groups = 1`, vLLM 0.22 `.venv/.../models/qwen3_5.py:756`) AND a DeepEP-ll/NIXL backend, while the backend is allgather_reducescatter. ∴ expert *e* is owned by DP engine `floor(e/64)` (rank0:0–63, rank1:64–127, rank2:128–191, rank3:192–255). **For pure-EP (tp1/dp4) the flattened `ep_rank == dp_rank == engine index`**, so the owning rank **is** the `/metrics` engine port v1 already enumerates (29520+floor(e/64)). (Note: `ep_rank == dp_rank` holds only because TP=1; the general flattened-EP rank is the TP×DP flatten — see arm-C note. Confirm the rank↔port mapping from the boot log, not a line-number cite.)
